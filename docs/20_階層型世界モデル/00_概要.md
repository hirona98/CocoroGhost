# 階層型世界モデル 概要

## 1. 位置づけ

本書は、CocoroGhost を「会話応答中心の記憶システム」から、
「自己像を持ち、観測・意思決定・行動・内省を循環させる人格システム」へ拡張するための仕様を定義する。

本書で固定する対象は、能力一覧ではなく次の2点である。

1. 階層制御（戦略/戦術/実行/反省）の責務分離
2. 能力追加時にも崩れない共通データ契約

## 2. 最終ゴール（人間模擬）

本拡張の最終ゴールは「人間らしい連続性を持つ AI 人格」の実現である。

必要な性質:

1. 長期連続性
   - 目標・関係性・価値観が日を跨いで継続する
2. 短期適応性
   - 直近の出来事に反応し、行動計画を更新できる
3. 自己整合性
   - 発話・行動・記憶（World Model）間の矛盾を減らし続ける
4. 説明可能性
   - なぜその行動を選んだかを goal/ticket/result/revision で追跡できる

## 3. 非ゴール

1. capability の固定一覧化
2. 旧アーキテクチャとの後方互換
3. capability 固有分岐を基盤ループへ直接埋め込む設計

## 4. 設計原則

1. World Model First
   - 行動前に世界状態を更新し、行動後も必ず反映する
2. Contract First
   - capability 追加より先に層間契約を固定する
3. Hierarchical Control
   - 戦略（長期意図）と戦術（次の1手）を分離する
4. Event Grounded
   - 行動結果・観測結果は events と wm_* に残し、次サイクルに再利用する
5. Fail Fast
   - adapter 未登録、schema 不一致、実行失敗は即失敗として確定する
6. No Fallback
   - 代替経路や暗黙デフォルトによる救済は入れない

## 5. 階層制御モデル

```text
[Observation Ingest]
    -> [World Model Update]
    -> [Strategic Layer]
    -> [Tactical Layer]
    -> [Execution Layer (Capability Adapter)]
    -> [Result Ingest]
    -> [World Model Update]
    -> [Reflection]
```

責務:

1. Observation Ingest
   - 永続 event / action_result / system trigger を正規化する
2. Strategic Layer
   - 長期目標（StrategicGoal）を生成・更新・停止する
3. Tactical Layer
   - active goal を単発実行単位（ActionTicket）へ分解する
4. Execution Layer
   - `capability_id + operation` を registry 解決し1回だけ実行する
5. Reflection
   - 実行結果差分を評価し、goal/belief を更新する

## 6. 固定データ契約

### 6.1 StrategicGoal

```json
{
  "goal_id": "uuid",
  "goal_type": "maintenance|social|task|exploration",
  "title": "string",
  "intent": "string",
  "priority": 0.0,
  "horizon_seconds": 86400,
  "success_criteria": ["string"],
  "constraints": ["string"],
  "status": "active|paused|done|dropped",
  "reason_code": null,
  "updated_at": "2026-02-15T12:00:00"
}
```

### 6.2 ActionTicket

```json
{
  "ticket_id": "uuid",
  "goal_id": "uuid",
  "capability_id": "string",
  "operation": "string",
  "input_payload": {},
  "preconditions": ["string"],
  "expected_effect": ["string"],
  "verify": ["string"],
  "status": "queued|running|succeeded|failed|cancelled|expired",
  "reason_code": null,
  "issued_at": "2026-02-15T12:00:00",
  "deadline_at": "2026-02-15T12:05:00"
}
```

### 6.3 CapabilityDescriptor（能力単位）

```json
{
  "capability_id": "string",
  "display_name": "string",
  "enabled": true,
  "version": "1",
  "metadata_json": {},
  "operations": ["string"]
}
```

### 6.4 CapabilityOperationDescriptor（operation 単位）

```json
{
  "capability_id": "string",
  "operation": "string",
  "input_schema_json": {},
  "result_schema_json": {},
  "effect_schema_json": {},
  "timeout_seconds": 30,
  "enabled": true
}
```

### 6.5 ActionResult

```json
{
  "result_id": "uuid",
  "ticket_id": "uuid",
  "status": "succeeded|failed",
  "observations": [],
  "effects": [],
  "error_message": null,
  "reason_code": null,
  "finished_at": "2026-02-15T12:00:03"
}
```

### 6.6 ObservationRecord

```json
{
  "observation_id": "uuid",
  "source_type": "event|action_result|system",
  "source_ref": "event:123",
  "summary": "string",
  "payload_json": {},
  "importance": 0.0,
  "observed_at": "2026-02-15T12:00:00"
}
```

### 6.7 LoopTrigger

```json
{
  "trigger_type": "event_created|action_result|periodic|startup",
  "event_id": 123,
  "result_id": "uuid",
  "created_at": "2026-02-15T12:00:00"
}
```

必須条件:

1. `trigger_type=event_created` のとき `event_id` 必須
2. `trigger_type=action_result` のとき `result_id` 必須
3. それ以外は付随ID省略可

## 7. World Model（最小層）

World Model は次の4層で保持する。

1. Entity
   - 人/場所/物/デバイス/Webリソース/自己状態
2. Relation
   - entity 間関係（located_in/owns/connected_to 等）
3. Belief
   - 命題 + 確信度 + 有効期間 + 根拠
4. Intent State
   - goal/ticket/result の進行状態

最小要件:

1. 矛盾する belief は上書きせず並存させる
2. すべての状態遷移に reason_code を残す
3. events と wm_* を双方向参照可能にする（link/revision）

## 8. ループ規約

1. Observe
   - 命令イベント（例: `vision.capture_request`）は入力に含めない
   - 永続 event / action_result / system trigger を ObservationRecord 化する
2. Strategize
   - 同一サイクルで同一 goal を複数回更新しない
3. Tacticalize
   - 1 goal あたり同一サイクルで新規 ticket は最大1件
   - precondition 未成立 ticket は `cancelled` を明示する
4. Execute
   - registry 解決 -> input 検証 -> adapter 1回実行 -> result/effect 検証
5. Reflect
   - failed 結果も必ず評価し、再計画要否を決める
6. action_result trigger
   - 同一 `result_id` での再起動は1回までに制限する

## 9. 既存基盤との統合方針

1. `/api/chat` 同期経路に重い処理を混ぜない
2. 自律ループは worker/periodic で実行する
3. 自律ループの結果は `events` に永続化し WritePlan に接続する
4. command event は非永続化（`event_id=0`）のまま維持する

## 10. `web_access` と `/api/chat` Web検索の分離

1. `/api/chat` の `reply_web_search_enabled`
   - 会話最終応答（L3）専用
2. `web_access` capability
   - 自律ループの ActionTicket として実行
3. 両者は設定・実行タイミング・永続化責務を共有しない

## 11. 拡張ルール（能力後付け）

新 capability 追加時に必要なのは次の4点のみ。

1. CapabilityDescriptor 登録
2. adapter 実装
3. operation schema 定義
4. effect の world model 反映規則定義

禁止事項:

1. `loop_runtime` への capability 固有 if 追加
2. `worker_handlers.py` への capability 個別分岐追加
3. fallback 専用経路の追加

## 12. 評価指標（人間模擬）

1. 目標継続率
   - goal が複数日維持される割合
2. 行動妥当率
   - expected_effect と実結果の一致率
3. 記憶整合率
   - 発話と wm_beliefs の矛盾率
4. 再計画速度
   - failed 後に次の実行可能 ticket を出すまでのサイクル数
5. 追跡可能率
   - goal/ticket/result/revision が根拠イベントへ辿れる割合
