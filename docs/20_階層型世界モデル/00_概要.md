# 階層型世界モデル 概要

## 1. 位置づけ

本書は、CocoroGhost を「会話中心の記憶システム」から、
「世界を観測し、目的を持って自発行動する人格システム」へ拡張するための概要である。

この文書で固定するのは capability 一覧ではない。
固定するのは、能力を後から追加できる **階層構造** と **共通契約** である。

## 2. 背景

現行設計は記憶品質と会話体感を優先しており、次の性質を持つ。

- `/api/chat` は同期経路で想起を確定し、SSE開始を優先する
- 重い更新は worker の非同期経路へ分離されている
- `meta_proactive` / `desktop_watch` / `vision.capture_request` など自発導線の土台はある

一方で、将来は次を前提とする。

- 五感（視覚以外を含む）
- 移動
- ネットワーク家電操作
- 新能力の継続的な後付け

この前提では、能力一覧を先に固定する方式は適合しない。
そのため、世界モデル中心の階層型アーキテクチャへ移行する。

## 3. 目的

1. 人間らしい「長期意図」と「短期反応」の両立
2. capability 後付けを前提にした拡張可能性
3. 既存の会話/記憶基盤と矛盾しない統合
4. 行動結果を必ず記録し、次の思考へ循環させる

## 4. 非ゴール

1. capability の固定一覧化
2. 旧アーキテクチャとの後方互換
3. 単発機能ごとの場当たり的な専用実装

## 5. 基本方針

1. World Model First
   - まず世界状態を更新し、その上で行動を決める
2. Contract First
   - capability 追加より先に、層間データ契約を固定する
3. Hierarchical Control
   - 戦略層（長期）と戦術層（短期）を分離する
4. Event Grounded
   - 観測結果と行動結果は `events` 系に残す（命令イベントの非永続化は許容する）
5. No Hidden Path
   - 例外経路を増やさず、同一ループで処理する

## 6. 全体構造

```text
[Observation Ingest]
    -> [World Model Update]
    -> [Strategic Layer]
    -> [Tactical Layer]
    -> [Execution Layer (Capability Adapter)]
    -> [Result Ingest]
    -> [World Model Update]
    -> [Reflection]
```

構成要素:

1. Observation Ingest
   - 会話、視覚、外部入力、実行結果を正規化して投入する
2. World Model Store
   - entity/relation/belief/goal/action を保持する
3. Strategic Layer
   - 時間軸の長い意図を更新する
4. Tactical Layer
   - 今実行する 1 手（ActionTicket）を選ぶ
5. Execution Layer
   - capability adapter を呼び、実世界へ作用する
6. Reflection
   - 実行結果を評価し、goal/belief を調整する

## 7. 階層定義

### 7.1 戦略層（Strategic Layer）

責務:

- 複数時間軸の目標を維持する
- 目標の優先順位を決める
- 成功条件を定義する

出力:

- `StrategicGoal` の更新

### 7.2 戦術層（Tactical Layer）

責務:

- Active な `StrategicGoal` を実行可能な 1 手へ分解する
- 実行前条件を確認する
- 実行後検証（verify）を定義する

出力:

- `ActionTicket`（単発実行単位）

### 7.3 実行層（Execution Layer）

責務:

- `capability_id` と `operation` から adapter を解決する
- 入力スキーマを検証する
- 実行結果を標準形式で返す

出力:

- `ActionResult`

## 8. 層間データ契約（固定）

capability 一覧は固定しないが、以下の契約は固定する。

### 8.1 StrategicGoal

```json
{
  "goal_id": "uuid",
  "title": "string",
  "intent": "string",
  "priority": 0.0,
  "horizon_seconds": 86400,
  "success_criteria": ["string"],
  "constraints": ["string"],
  "status": "active|paused|done|dropped",
  "updated_at": "2026-02-15T12:00:00"
}
```

### 8.2 ActionTicket

```json
{
  "ticket_id": "uuid",
  "goal_id": "uuid",
  "capability_id": "string",
  "operation": "string",
  "input_payload": {},
  "preconditions": ["string"],
  "expected_effect": ["string"],
  "verify": ["string"],
  "issued_at": "2026-02-15T12:00:00",
  "deadline_at": "2026-02-15T12:05:00"
}
```

### 8.3 CapabilityDescriptor

```json
{
  "capability_id": "string",
  "operations": ["string"],
  "input_schema_json": {},
  "result_schema_json": {},
  "effect_schema_json": {},
  "enabled": true
}
```

### 8.4 ActionResult

```json
{
  "ticket_id": "uuid",
  "status": "succeeded|failed",
  "observations": [],
  "effects": [],
  "error_message": null,
  "finished_at": "2026-02-15T12:00:03"
}
```

## 9. World Model（最小モデル）

World Model は次の 4 層で持つ。

1. Entity
   - 人、場所、物、デバイス、Webリソース、自己状態
2. Relation
   - entity 間の関係（located_in, owns, connected_to など）
3. Belief
   - 命題と確信度、時間範囲
4. Intent State
   - goal と action の進行状態

最小要件:

- belief は `confidence` と `updated_at` を必須とする
- 衝突する belief は上書きせず並存させる
- goal/action との因果リンクを保存する

## 10. 実行ループ

### 10.1 観測ループ

入力:

- chat/notification/reminder/desktop_watch/vision_detail/meta_proactive
- capability 実行結果

処理:

- 正規化 -> world model 更新 -> 監査ログ記録

注記:

- `vision.capture_request` のような「命令イベント」は、既存方針どおり非永続化で扱ってよい
- 命令の結果として得られた観測/行動結果は、`events` へ永続化して次の想起へ流す

### 10.2 戦略ループ

処理:

- world model から active goal を生成/更新
- 優先度と horizon を再計算

出力:

- `StrategicGoal` 差分

### 10.3 戦術ループ

処理:

- active goal から `ActionTicket` を発行
- 実行可能性を評価

出力:

- 実行キュー（ticket）

### 10.4 実行ループ

処理:

- `ActionTicket` を adapter にディスパッチ
- `ActionResult` を返却

### 10.5 反省ループ

処理:

- `ActionResult` で belief/goal/action を更新
- 必要に応じて対話メッセージを生成

## 11. 既存システムとの統合方針

1. `/api/chat` の同期経路は維持する
   - 自律ループの重い処理を混ぜない
2. 自律ループは worker/periodic 系に統合する
3. 自発メッセージは既存の events/stream 配信導線へ統合する
4. 行動結果・観測結果・反省は記録し、WritePlan 連携可能な形にする

### 11.1 Webアクセス能力と `/api/chat` Web検索の関係

- `web_access` は階層型世界モデルの実行層（capability adapter）で実行する能力とする
- `/api/chat` の `reply_web_search_enabled` は、会話最終応答（L3）専用の既存フラグとして維持する
- 記憶検索経路や自律実行経路を、`/api/chat` のWeb検索プラグイン制御と混同しない

## 12. 最初の capability（Webアクセス）

最初の実装対象は `web_access` とする。
ただし専用設計にはしない。`CapabilityDescriptor` に 1 件追加するだけで扱う。

注記:

- ここでいう `web_access` は自律実行能力であり、`/api/chat` 最終応答のWeb検索設定とは別物である

想定 operation:

1. `search`
2. `open_url`
3. `extract_structured`

想定 effect:

- world model への新規 belief 追加
- 既存 belief の確信度更新
- 対話で共有すべき要点の生成

## 13. 拡張ルール（能力後付け）

新 capability 追加時に必須なのは次の 4 点だけとする。

1. `CapabilityDescriptor` 登録
2. adapter 実装
3. operation ごとの schema 定義
4. effect の world model 反映規則

戦略層/戦術層の本体ロジックには、能力固有分岐を持ち込まない。

## 14. 観測・評価

評価は「人間らしさ」と「運用性」の両面で行う。

1. 目標継続性
   - goal が複数ターン・複数日で維持されるか
2. 文脈一貫性
   - world model と対話内容の矛盾率
3. 行動妥当性
   - expected_effect と結果の一致率
4. 修正速度
   - 失敗後の再計画までのサイクル数
5. 透明性
   - ticket/result/belief 更新理由の追跡可能性

## 15. 実装フェーズ

1. Phase 0: 契約固定
   - `StrategicGoal` / `ActionTicket` / `CapabilityDescriptor` / `ActionResult`
2. Phase 1: 階層型世界モデル骨格
   - 戦略/戦術/実行/反省ループの最小実装
3. Phase 2: `web_access` 追加
   - operation 3種を登録して end-to-end で実行
4. Phase 3: 仕様へ分割
   - 感覚、移動、家電制御を個別文書で追加
