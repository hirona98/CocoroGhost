# 階層型世界モデル 整合化修正計画

## 1. 目的

本書は、階層型世界モデルの仕様（`00_概要.md`〜`05_web_access.md`）を正として、
実装をあるべき姿へ揃えるための実施計画を定義する。

## 2. 前提

1. 正はドキュメント側（`docs/20_階層型世界モデル/`）とする
2. 後方互換をつけない
3. fallback を導入しない
4. 単一ユーザー運用を前提とする

## 3. 現在の主要不一致

1. `wm_*` テーブル契約（列名/制約/インデックス）が仕様と不一致
2. `ObservationRecord` 契約（UUID/summary/importance/observed_at）が未実装
3. `world_model_update_request` の生成/適用フローが未実装
4. adapter が DB 更新しており、Adapter 規約と不一致
5. `action_result` trigger は仕様化済みだが、投入経路が未接続
6. capability schema 検証が Draft 2020-12 契約より弱い
7. `StrategicGoal` の `goal_type` / `reason_code` が永続化されていない

## 4. 修正方針

1. ドキュメントを変更して現行実装へ合わせる方針は取らない
2. 先に契約層（モデル/I/F）を揃えてから、実行経路を揃える
3. 大きい差分は段階投入し、フェーズごとに動作確認する
4. DB スキーマは `memory_*.db` の再作成前提で進める

## 5. 実施フェーズ

### Phase 1: ループ契約の接続

目的:

1. trigger 契約とループ入口を仕様に揃える

実施:

1. `action_result` trigger の投入経路を追加する
2. 同一 `result_id` 多重投入禁止を維持する
3. `run_autonomy_cycle` の trigger 正規化を仕様準拠で固定する

対象:

1. `cocoro_ghost/autonomy/loop_runtime.py`
2. `cocoro_ghost/autonomy/scheduler.py`

### Phase 2: World Model 永続化契約の再実装

目的:

1. `02_world_model.md` の最小テーブル仕様へ実装を一致させる

実施:

1. `memory_models.py` の `Wm*` モデルを仕様の列構成へ更新する
2. `db.py` の `wm_*` インデックスを仕様へ合わせる
3. `world_model_store.py` の I/F を新スキーマへ合わせる
4. `StrategicGoal.reason_code` / `goal_type` を永続化する

対象:

1. `cocoro_ghost/memory_models.py`
2. `cocoro_ghost/db.py`
3. `cocoro_ghost/autonomy/world_model_store.py`

### Phase 3: world_model_update_request 導入

目的:

1. Observe/Reflect の更新I/Fを仕様通りに分離する

実施:

1. `world_model_update_request` 契約をコード化する
2. Observe で request を生成する
3. Reflect で request を transaction 適用する
4. `revisions` 記録を request 適用に統合する

対象:

1. `cocoro_ghost/autonomy/contracts.py`
2. `cocoro_ghost/autonomy/loop_runtime.py`
3. `cocoro_ghost/autonomy/world_model_store.py`

### Phase 4: Capability Registry 検証強化

目的:

1. schema 検証を `05_web_access.md` と `03_capability_registry.md` に一致させる

実施:

1. Draft 2020-12 準拠の検証へ置き換える
2. `format` / `const` / `anyOf` / `minimum` 等を検証対象に含める
3. `capability_bootstrap.py` の descriptor を仕様 JSON Schema と一致させる

対象:

1. `cocoro_ghost/autonomy/capability_registry.py`
2. `cocoro_ghost/autonomy/capability_bootstrap.py`

### Phase 5: Adapter 責務分離

目的:

1. adapter から DB 更新責務を除去し、Execute/Reflect 層へ戻す

実施:

1. adapter は external action 実行 + `result_payload/effects` 返却のみとする
2. result event 永続化と WritePlan ジョブ投入は `loop_runtime.py` へ集約する
3. `speak` / `web_access` 両 adapter を規約準拠へ変更する

対象:

1. `cocoro_ghost/autonomy/capability_adapters/speak.py`
2. `cocoro_ghost/autonomy/capability_adapters/web_access.py`
3. `cocoro_ghost/autonomy/loop_runtime.py`

### Phase 6: Event/WritePlan 連携の整合化

目的:

1. result event と WritePlan 連携順序を仕様へ揃える

実施:

1. result event 保存 -> 3ジョブ投入の順序を固定する
2. `action_result` 起点サイクルを接続する
3. command event 非永続（`event_id=0`）規約を維持する

対象:

1. `cocoro_ghost/autonomy/loop_runtime.py`
2. `cocoro_ghost/autonomy/scheduler.py`
3. `cocoro_ghost/memory/_jobs_mixin.py`

### Phase 7: 整合性確認

目的:

1. 仕様と実装の一致を最終確認する

実施:

1. trigger / reason_code / schema 検証 / WritePlan 連携の動作確認
2. `docs/20_階層型世界モデル/*.md` の参照と契約名の整合を確認
3. 仕様差分が残っていないことをレビューで確認する

## 6. 実装順序

2. Phase 2

4. Phase 4

6. Phase 6

## 7. 完了条件

1. 第3章の不一致 1〜7 がすべて解消されている
2. `loop_runtime` に capability 固有分岐を追加していない
3. adapter が DB を直接更新していない
4. docs とコードの再レビューで矛盾が検出されない
