# 階層型世界モデル 基盤ループ

## 1. 文書の目的

本書は、階層型世界モデルの基盤ループ
（Observe / Strategize / Tacticalize / Execute / Reflect）を、
実装に直結する粒度で定義する。

この文書を読めば、次が分かる状態を目標とする。

1. どんな機能を作るか
2. どのファイルに何を実装するか
3. 何を満たせば Phase 1 完了か

## 2. Phase 6 で実装する機能（本書で仕様固定するMVP）

## 2.1 必須機能

1. ループ実行器
   - 5ループを順番に1サイクル実行する `run_cycle(trigger)` を実装する
2. トリガ投入
   - `event_created` / `action_result` / `periodic` / `startup` の4種類でサイクルを起動できる
3. Goal/Ticket/Result の状態管理
   - `StrategicGoal` / `ActionTicket` / `ActionResult` の状態遷移を実装する
4. 実行層のディスパッチ
   - `capability_id` + `operation` で adapter を解決し、`ActionResult` を確定する
5. 自発発話の統合
   - `operation=speak` を通常の `ActionTicket` と同経路で実行し、
     `events.source="meta_proactive"` として保存・配信する
6. 既存ランタイム統合
   - `/api/chat` 同期経路には混ぜず、worker/periodic 経路だけで動かす

## 2.2 このフェーズでやらないこと

1. World Model 永続化スキーマの詳細（Phase 2）
2. Capability Registry 詳細仕様（Phase 3）
3. `web_access` 等の能力個別I/O（Phase 5）

## 3. ランタイム配置

## 3.1 実行場所

1. ループ本体: worker job (`jobs.kind`) として実行
2. periodic: ループ本体を直接実行せず、job投入だけ行う
3. `/api/chat`: 既存同期フローを維持（変更しない）

## 3.2 サイクルの入口

```text
event保存 / periodic tick / startup
    -> jobs(kind="autonomy_cycle", payload={trigger...}) を投入
    -> worker が autonomy_cycle を実行
    -> run_cycle(trigger)
```

## 3.3 実行頻度

1. event起点: event保存ごとに起動可
2. periodic起点: 1秒周期で起動可（実行中は重複起動しない）
3. startup起点: 起動時に1回起動

## 4. データ契約（基盤ループで固定）

`StrategicGoal` / `ActionTicket` / `ActionResult` のスキーマ本体は
`00_概要.md` の定義を正とする。

基盤ループでは、次の追加契約を固定する。

## 4.1 LoopTrigger

```json
{
  "trigger_type": "event_created|action_result|periodic|startup",
  "event_id": 123,
  "result_ticket_id": "uuid",
  "created_at": "2026-02-15T12:00:00"
}
```

ルール:

1. `trigger_type=event_created` のとき `event_id` 必須
2. `trigger_type=action_result` のとき `result_ticket_id` 必須
3. それ以外では付随IDは省略可

## 4.2 ObservationRecord

```json
{
  "observation_id": "uuid",
  "source": "event|action_result",
  "source_id": "event:123",
  "summary": "string",
  "importance": 0.0,
  "observed_at": "2026-02-15T12:00:00"
}
```

## 4.3 ReasonCode（状態遷移理由）

状態遷移時は `reason_code` を必須保存する。

1. `goal_precondition_missing`
2. `goal_success_criteria_met`
3. `goal_dropped_by_policy`
4. `ticket_precondition_failed`
5. `ticket_deadline_expired`
6. `execute_adapter_not_found`
7. `execute_schema_invalid`
8. `execute_runtime_error`
9. `reflect_replan_required`

## 5. ループ仕様（実装単位）

## 5.1 Observe

入力:

1. 新規 `events`（chat/notification/reminder/desktop_watch/vision_detail/meta_proactive）
2. 実行結果 `ActionResult`

処理:

1. トリガを `ObservationRecord` に正規化
2. 反省対象の `ActionResult` があれば観測に統合
3. World Model 反映リクエストを作成

出力:

1. `observations[]`
2. `world_model_update_requests[]`

実装ルール:

1. 命令イベント（例: `vision.capture_request`）は観測対象に含めない
2. 永続イベントだけを入力に使う

## 5.2 Strategize

入力:

1. `observations[]`
2. 現在の `StrategicGoal` 一覧（active/paused）

処理:

1. 目標生成または更新
2. 優先度計算
3. 状態更新（active/paused/done/dropped）

出力:

1. 更新済み `StrategicGoal[]`
2. `active_goal_ids[]`

実装ルール:

1. 同一サイクルで同一goalを複数回更新しない
2. 目標更新は `reason_code` を必須保存

## 5.3 Tacticalize

入力:

1. `active_goal_ids[]`
2. capability可用状態
3. 既存 `ActionTicket`（queued/running）

処理:

1. goalごとに ticket 候補を生成
2. preconditions を評価
3. 実行可能 ticket を `queued` で確定

出力:

1. 新規 `ActionTicket[]`
2. 実行対象 `queued_ticket_ids[]`

実装ルール:

1. 1 goal あたり同一サイクルで新規 ticket は最大1件
2. precondition未成立は `queued` 放置せず `cancelled` か `paused` を明示

## 5.4 Execute

入力:

1. `queued_ticket_ids[]`
2. capability adapter

処理:

1. ticket を `running` へ遷移
2. adapter 解決（`capability_id` + `operation`）
3. `input_payload` schema 検証
4. adapter 実行
5. `ActionResult` 確定、ticket を `succeeded|failed` に更新

出力:

1. `ActionResult[]`

実装ルール:

1. 未登録adapterは即 `failed`（`reason_code=execute_adapter_not_found`）
2. schema不一致は即 `failed`（`reason_code=execute_schema_invalid`）
3. fallback は行わない

## 5.5 Reflect

入力:

1. `ActionResult[]`
2. 関連 `StrategicGoal` / `ActionTicket`

処理:

1. expected_effect と結果差分を評価
2. goal/ticket 状態更新
3. 再計画要否を判定

出力:

1. goal/ticket 更新差分
2. 次サイクル起動要否

実装ルール:

1. `failed` 結果は必ず反省処理を通す
2. 反省結果は次サイクルで再利用できる形で保存する

## 6. 状態遷移

## 6.1 StrategicGoal

1. `active -> paused`: 前提不足（`goal_precondition_missing`）
2. `active -> done`: success_criteria達成（`goal_success_criteria_met`）
3. `active -> dropped`: 方針上の破棄（`goal_dropped_by_policy`）
4. `paused -> active`: 再開条件成立

## 6.2 ActionTicket

1. `queued -> running`: 実行開始
2. `running -> succeeded`: 実行成功
3. `running -> failed`: 実行失敗（adapter/schema/runtime）
4. `queued -> cancelled`: 上位goal停止またはprecondition未成立
5. `queued -> expired`: deadline超過（`ticket_deadline_expired`）

## 8. 擬似コード

```python
def run_cycle(trigger: LoopTrigger) -> None:
    observations = observe(trigger)
    goals = strategize(observations)
    tickets = tacticalize(goals)
    results = execute(tickets)
    reflect(results)
```

## 9. 自発発話（speak）の扱い

1. 自発発話は `operation=speak` の `ActionTicket` として発行する
2. 実行結果は `events.source="meta_proactive"` で保存する
3. UI配信は既存 `/api/events/stream` へ publish する
4. 発話専用の別経路は作らない

## 10. エラー方針

1. fail-fast を正とする
2. adapter未登録・schema不一致・実行例外は即 `failed`
3. fallback は入れない
4. `ActionResult.error_message` と `reason_code` を必ず保存する

## 11. 観測項目（最低限）

1. `autonomy_cycle` 実行回数（trigger_type別）
2. goal状態遷移件数（状態別）
3. ticket状態遷移件数（状態別）
4. ループ別処理時間（Observe/Strategize/Tacticalize/Execute/Reflect）
5. 自発発話件数（`operation=speak`）
6. 失敗件数（reason_code別）
