# 階層型世界モデル 基盤ループ

## 1. 目的

本書は、階層型世界モデルの基盤ループ
（観測/戦略/戦術/実行/反省）を仕様として定義する。

本書で決めるのは「ループの振る舞い」と「層間の状態遷移」であり、
能力個別仕様（`web_access` 等）は扱わない。

## 2. スコープ

対象:

1. 5ループの責務
2. ループ起動条件
3. `StrategicGoal` / `ActionTicket` / `ActionResult` の状態遷移
4. 自発発話を含む行動決定と実行の流れ
5. 既存 `/api/chat`・worker・periodic との接続方針

非対象:

1. World Model 永続化スキーマの詳細
2. Capability Registry の詳細
3. 能力個別の I/O 詳細

## 3. 前提

1. `/api/chat` の同期経路には重い処理を混ぜない
2. 基盤ループは worker/periodic 側で動かす
3. 命令イベント（例: `vision.capture_request`）は非永続化でよい
4. 観測結果・行動結果は `events` に永続化する
5. `/api/chat` 最終応答のWeb検索フラグと自律能力は別制御とする

## 4. ループ構成

## 4.1 観測ループ（Observe）

入力:

1. `events` の新規行（chat/notification/reminder/desktop_watch/vision_detail/meta_proactive）
2. capability 実行結果（`ActionResult`）

処理:

1. 入力を正規化して Observation レコード化
2. World Model 更新リクエストを作成
3. 更新対象（entity/relation/belief/intent）をキューへ投入

出力:

1. Observation 更新ログ
2. World Model 更新キュー

## 4.2 戦略ループ（Strategize）

入力:

1. World Model の最新状態
2. active/paused の `StrategicGoal`
3. 直近の失敗/成功履歴

処理:

1. 目標候補を生成または更新
2. 優先度を再計算
3. `status` を `active|paused|done|dropped` で更新

出力:

1. `StrategicGoal` 差分
2. 次サイクルで処理すべき goal 一覧

## 4.3 戦術ループ（Tacticalize）

入力:

1. `StrategicGoal`（`status=active`）
2. Capability 可用状態
3. 実行中/待機中の `ActionTicket`

処理:

1. 1 goal につき実行候補 ticket を生成
2. 前提条件（preconditions）を評価
3. 実行可能 ticket のみ `queued` として確定

出力:

1. `ActionTicket`
2. 実行キュー

## 4.4 実行ループ（Execute）

入力:

1. `ActionTicket`（`status=queued`）
2. Capability adapter

処理:

1. `capability_id` と `operation` で adapter を解決
2. `input_payload` を schema 検証
3. adapter 実行
4. `ActionResult` を確定

出力:

1. `ActionResult`
2. 実行ログ
3. 必要時のユーザー向けメッセージ

注記:

- 自発発話は `operation=speak` の通常 ticket として処理する
- 発話だけを特別経路にしない

## 4.5 反省ループ（Reflect）

入力:

1. `ActionResult`
2. 関連 `StrategicGoal` / `ActionTicket`
3. 実行時観測

処理:

1. 期待効果との差分評価
2. goal/ticket の状態更新
3. 次手（再実行/方針変更/終了）判定

出力:

1. goal/ticket 更新
2. World Model への反映要求
3. 反省ログ

## 5. 状態遷移

## 5.1 StrategicGoal

状態:

1. `active`
2. `paused`
3. `done`
4. `dropped`

遷移:

1. `active -> paused`: 前提不足または明示停止
2. `active -> done`: success_criteria 達成
3. `active -> dropped`: 目標破棄
4. `paused -> active`: 再開条件成立

## 5.2 ActionTicket

状態:

1. `queued`
2. `running`
3. `succeeded`
4. `failed`
5. `cancelled`
6. `expired`

遷移:

1. `queued -> running`: 実行ループが取得
2. `running -> succeeded`: 実行成功
3. `running -> failed`: 実行失敗
4. `queued -> cancelled`: 上位goal停止
5. `queued -> expired`: deadline 超過

注記:

- fallback は行わない
- `failed` は反省ループで次手を再計画する

## 6. ループ起動条件

## 6.1 イベント駆動

トリガ:

1. 新規 `events` 永続化
2. `ActionResult` 確定

対象ループ:

1. 観測ループ
2. 反省ループ（`ActionResult` 時）

## 6.2 周期駆動

トリガ:

1. periodic tick

対象ループ:

1. 戦略ループ
2. 戦術ループ
3. 実行ループ

## 6.3 起動時

トリガ:

1. アプリ起動

対象ループ:

1. 未処理 ticket 再評価
2. active goal 復元

## 7. 既存ランタイムとの接続

## 7.1 配置方針

1. 重いループ処理は worker 側で実行する
2. periodic は「起動トリガ投入」に限定する
3. `/api/chat` は既存の同期フローを維持する

## 7.2 競合制御

1. 同一 `goal_id` への同時更新は禁止する
2. `ticket_id` の実行は単一実行とする
3. 更新競合時は fail-fast で再試行キューへ戻す

## 7.3 発話の接続

1. `operation=speak` の `ActionTicket` を実行する
2. 結果を `events.source="meta_proactive"` として保存する
3. `events/stream` 経由で配信する

## 8. エラー方針

1. schema 不一致は即失敗（`failed`）
2. adapter 未登録は即失敗（`failed`）
3. precondition 未成立は `queued` 維持ではなく `paused` または `cancelled` を明示する
4. 失敗理由は必ず構造化して保存する

## 9. 観測項目

必須メトリクス:

1. goal 更新件数（状態別）
2. ticket 実行件数（状態別）
3. ループ処理時間（Observe/Strategize/Tacticalize/Execute/Reflect）
4. 発話アクション件数（自発）
5. 失敗理由の分類件数

必須ログ:

1. ticket 発行理由
2. ticket 実行結果
3. goal 状態遷移理由

## 10. 受け入れ条件

1. 観測入力から goal 更新まで 1 サイクル通る
2. active goal から ticket 発行・実行・反省まで 1 サイクル通る
3. `operation=speak` が通常 ticket と同じ経路で実行される
4. `/api/chat` 同期経路へ追加待ち時間を発生させない
5. 命令イベント非永続化と結果イベント永続化が両立する

## 11. 次フェーズへの引き継ぎ

1. World Model 仕様へ必要なI/F一覧
2. Capability Registry で必要な検証項目
3. Event/WritePlan連携で必要な保存ルール

本書で確定した状態遷移と起動条件は、後続フェーズで変更しない。
